# Архитектура интеллектуальной рекомендательной системы обучения (Математика)

## 1. Общее описание (Overview)
Система представляет собой мультиагентный конвейер для построения индивидуального плана обучения математике. 
Ключевая логика: отказ от статических путей в пользу динамического графа знаний и генерации заданий. Система работает асинхронно, метрика "время решения" (`expected_time`) не используется, заменена на временные метки (`timestamp`) для расчета кривой забывания.

---

## 2. Структуры данных (Схемы)

### 2.1. Метаданные задания (Item Schema)
Каждое задание в базе или сгенерированное на лету должно соответствовать следующей структуре:
* `task_id`: String (UUID)
* `primary_kcs`: List[String] - ID напрямую проверяемых навыков (узлы графа знаний).
* `secondary_kcs`: List[String] - ID скрытых пререквизитов.
* **IRT параметры (Item Response Theory):**
  * `difficulty`: Float (Параметр $b$)
  * `discrimination`: Float (Параметр $a$)
  * `guessing_probability`: Float (Параметр $c$)
* `distractors_map`: Dict[String, String] - Маппинг неправильных ответов на ID типичных ошибок (Misconception IDs). Пример: `{"-28": "M_loss_of_sign_in_square"}`.
* `scaffolding_steps`: List[String] - Многоуровневые подсказки от намека до прямого алгоритма.

### 2.2. Профиль ученика (Student Model Schema)
Вектор состояния знаний и поведенческих паттернов.
* `student_id`: String (UUID)
* **Вектор знаний (Knowledge State):**
  * `mastery_probabilities`: Dict[String, Float] - Вероятность (0.0 - 1.0) освоения каждого KC (микро-навыка).
  * `last_practiced`: Dict[String, Timestamp] - Метка времени последнего обращения к каждому KC для функции забывания (Decay).
* **Поведенческие метрики:**
  * `guessing_rate`: Float - Склонность к слепому перебору.
  * `hint_dependence`: Float - Частота запросов подсказок до попытки решения.
* **История (Trace History):**
  * `interactions`: List[Dict] - Лог попыток (`task_id`, `is_correct`, `hints_used`, `timestamp`, `misconception_triggered`).

## 3. Архитектура пайплайна (Core Agents)

Система разделена на 4 взаимодействующих компонента.

### Агент 1: State Tracker (Модуль обновления состояний)
* **Вход:** Лог последнего ответа ученика.
* **Логика:** Использует Байесовское прослеживание знаний (BKT) или DKT.
* **Действие:** Пересчитывает `mastery_probabilities` в профиле ученика с учетом `timestamp` (штраф за забывание), использования подсказок и выявленных через `distractors_map` ошибок.

### Агент 2: Candidate Retrieval (Модуль поиска по графу)
* **Вход:** Обновленный профиль ученика, Целевой узел графа.
* **Логика:** Интеграция с графовой базой знаний (рекомендуется использование фреймворков вроде LightRAG для эффективного извлечения зависимостей пререквизитов). 
* **Действие:** Находит подграф навыков в "Зоне ближайшего развития" (базовые навыки > 0.60, текущие < 0.95). Возвращает пул из ~50 подходящих концептов/задач. Если State Tracker выявил "дыру" в знаниях, алгоритм принудительно спускается на уровень ниже по ребрам графа.

### Агент 3: Ranking Model (Педагогический модуль)
* **Вход:** Пул из 50 задач-кандидатов.
* **Логика:** Item Response Theory (IRT). Сопоставляет `difficulty` задачи с `mastery_probability` ученика.
* **Действие:** Отсеивает слишком легкие и слишком сложные задачи. Выбирает задачу с вероятностью успешного решения в диапазоне 60-75%.

### Агент 4: Math Generation & Scaffolding (Модуль генерации)
* **Вход:** Выбранный концепт (KC) и требуемый уровень сложности.
* **Логика:** LLM-агент для динамической генерации математических задач.
* **Действие:** Если статический банк исчерпан или требуется уникальная задача для защиты от списывания, генерирует задачу с нужными `distractors_map`. В процессе решения отвечает за подачу контекстных подсказок (scaffolding) в режиме диалога на основе допущенной ошибки.

---

## 4. Жизненный цикл события (Event Loop)
1. Ученик отправляет ответ или запрашивает подсказку.
2. `State Tracker` фиксирует `timestamp` и обновляет вероятности узлов в профиле.
3. `Candidate Retrieval` опрашивает граф знаний и строит маршрут (вперед к цели или назад к пререквизитам).
4. `Ranking Model` выбирает оптимальную сложность следующего шага.
5. `Math Generation Agent` отдает готовую или генерирует новую задачу в интерфейс.
