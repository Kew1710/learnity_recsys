✓ Сильные стороны Что сделано правильно
🧠 BKT с decay — правильный старт
Классический BKT (L0, T, S, G) с экспоненциальным decay отдельно от mastery_stored — это именно то, что нужно на MVP. Не нужно нейросетей без данных. DKT/DKVMN потребует тысяч студентов, а BKT работает с единицами. Разделение mastery_stored / mastery_effective — грамотная инженерная абстракция, позволяющая менять функцию decay без миграции данных.
interpretable data-efficient
🗺️ ZPD на графе с assumed mastery
grade_floor = student_grade - 2 с автоматическим assumed mastery для пререквизитов ниже floor — это продуманное решение, избегающее проблемы «ученик застрял из-за непроверенных древних тем». STRONG_PREREQ = 0.5 позволяет гибко строить граф, не блокируя студентов слабыми зависимостями. Чистая функция без side effects — легко тестируется.
pedagogically sound testable
📐 Таксономия по реальной программе
142 KC построены по реальному школьному перечню тем 5–11 класс с корректными grade_introduced из CSV-таксономии. Поле subject (algebra/geometry/arithmetic/statistics) позволяет будущему ранжировщику балансировать темы. difficulty_base как float — правильный прокси до IRT-калибровки.
curriculum-aligned extensible
🔌 Чёткое разделение сервисов (SRP)
Graph /zpd принимает mastery как параметр, не ходит в profile сам — clean SRP. Profile пишет mastery, все читают. Kafka для холодного пути позволяет replay и независимое масштабирование. Архитектура позволит добавить Retrieval между Graph и Ranking без переписывания существующих сервисов.
loose coupling event-driven
🧪 Качество тестового покрытия
73 теста: unit (чистые функции BKT, ZPD) + integration (реальный PG с savepoint rollback, реальный Neo4j). Savepoint-подход в integration tests — правильно: изолирует состояние без пересоздания БД. Fixture с module scope для Neo4j seed — хорошо для скорости. Тесты обеспечивают контракт на все граничные случаи ZPD (assumed mastery, weak prereqs, grade filter).
unit + integration real DB
📊 Схема данных с частичными ответами
score: float [0.0–1.0] вместо бинарного correct — позволяет гранулярное BKT-обновление. primary_kcs / secondary_kcs с разными весами (1.0/0.3) — корректно отражает реальную педагогику, где задание тренирует несколько тем. distractors_map → misconception_id — задел для диагностики типичных ошибок.
non-binary feedback multi-KC tasks
⚠ Риски Слабые места
🔢 BKT параметры — одинаковые для всех KC
В текущей реализации BKT параметры (P_T, P_S, P_G) либо глобальные, либо per-student. Но в реальности «вероятность выучить за одну попытку» у тригонометрии и у натуральных чисел кардинально разная. Без per-KC параметров BKT недо- или переоценивает прогресс. Нужна инициализация p_transit / p_slip / p_guess на уровне KC-ноды в Neo4j.
model accuracy risk
📉 Decay функция — недостаточно реалистична
0.5 ** (days / half_life) с фиксированным half_life=30 — грубое приближение. Кривая Эббингауза нелинейна, а half_life сильно варьируется: процедурные навыки (умножение) забываются медленнее концептуальных (производная). Также отсутствует эффект спейсинга — повторение на следующий день не должно давать тот же прирост, что через неделю. Рекомендую хотя бы per-KC half_life.
forgetting model
🎲 Нет exploration — чистый exploitation
IRT + ZPD выбирают «оптимальное» задание детерминированно. Это exploitation без exploration: система никогда не узнает, что задание с difficulty=0.6 плохо откалибровано, если всегда выбирает difficulty=0.5. Thompson Sampling или ε-greedy обязателен после накопления базовых данных. Иначе система сходится в локальный оптимум и перестаёт «видеть» новые задания.
exploitation trap
❄️ Cold Start — mastery=0.5 для старых классов рискован
Инициализация mastery=0.5 для KC ниже текущего класса — смелое допущение. Ученик 8-го класса с реальным mastery=0.1 по дробям получит ZPD в 8-м классе, при этом не освоив base. 10–15 диагностических вопросов в Placement Assessment явно недостаточно для покрытия 30+ KC из предыдущих классов. Нужна адаптивная диагностика: CAT (Computerized Adaptive Testing) хотя бы на 20–30 вопросов с IRT-выбором следующего вопроса.
cold start quality
🔗 IRT параметры — нет плана калибровки
Для синтетического seed задания получат irt=None или прокси difficulty_base. Но IRT 2PL/3PL нужно калибровать на реальных ответах (минимум 200–300 студентов per item). Пока их нет — нужен явный fallback: ранжировать по difficulty_base, не претендуя на IRT. Иначе система даёт иллюзию точности там, где данных нет.
calibration gap
📏 Граф — нет механизма обновления
142 KC с 253 рёбрами — хорошо для MVP. Но граф статичен: нет процесса валидации корректности рёбер, нет versioning, нет механизма добавления новых KC при расширении банка заданий. Что происходит если задание тегировано KC, которой нет в графе? Нужна хотя бы schema validation при seed и проверка orphan KC при добавлении задания.
graph drift risk
📊 Evaluation pipeline полностью отсутствует
Нет метрик обучения, нет offline-evaluation рекомендаций, нет способа понять «система помогает?». Это блокер для production: без eval loop невозможно итерировать на модели. Подробнее — в разделе Evaluation ниже.
critical gap blocks production
🔧 Улучшения Что стоит изменить — по приоритету
1
Добавить per-KC параметры BKT в Neo4j
Сейчас P(T), P(S), P(G) либо глобальные либо не используются. Добавить p_transit_default, p_slip_default, p_guess_default на KC-ноду. Инициализировать разумными prior'ами по subject (geometry сложнее arithmetic → ниже p_transit). Это улучшит качество BKT без новых данных.
Effort: S
2
Explicit IRT fallback — difficulty_base как прокси
В Ranking сервисе явно разделить два режима: irt_mode (если irt != None) и difficulty_mode (fallback на difficulty_base). Логировать режим в каждом запросе. Не смешивать — иначе невозможно понять что работает.
Effort: S
3
Минимальный eval: логировать mastery дельту
После каждого BKT update логировать mastery_before, mastery_after, kc_id, is_correct в Kafka topic mastery_events. Это даст материал для offline evaluation без дополнительной работы. Позже — агрегировать в learning gain метрику.
Effort: S
4
Thompson Sampling в Ranking — запланировать архитектурно
Не обязательно реализовывать сразу, но заложить в контракт Ranking: принимать exploration_rate: float (дефолт 0.0). При exploration_rate > 0 — с вероятностью ε выбирать случайное задание из ZPD вместо оптимального. Это позволит включить exploration одной конфигурацией.
Effort: M
5
Decay half_life — per-KC или per-subject
Добавить half_life_days на KC-ноду (или хотя бы на subject). Разумные дефолты: arithmetic=90d, algebra=45d, geometry=60d, statistics=30d. Это небольшое изменение, но значительно улучшает реалистичность decay без усложнения кода.
Effort: S
6
Placement Assessment — адаптивный выбор вопросов
Вместо фиксированных 10–15 вопросов: начать с KC среднего grade ученика, после ответа выбирать следующий вопрос по результату (правильно → сложнее, неправильно → проще). Это CAT в простейшем виде — 20 вопросов дадут точность 30-вопросного фиксированного теста. Реализуется на ZPD + простой binary search по difficulty.
Effort: M
7
Graph versioning + orphan KC validation
Добавить graph_version в Neo4j метаданные. При добавлении задания — валидировать что все KC из primary_kcs существуют в графе. При изменении графа — проверять что не возникает orphan KC (задания без KC в графе). Можно реализовать как pre-commit check в seed скрипте.
Effort: M
8
Interaction log — добавить поле recommendation_source
В таблицу interactions добавить recommendation_source: "zpd" | "placement" | "exploration" | "manual". Без этого невозможно будет разделить качество рекомендаций по типу. Нулевая стоимость сейчас, огромная ценность при анализе.
Effort: XS

Критичный пробел: Evaluation Pipeline

Как измерять что система работает — конкретные метрики и методология

Метрика 1 · Онлайн
Learning Gain per Session
Средний прирост mastery_effective за одну сессию. Ключевой сигнал что рекомендации работают. Агрегировать per-KC и per-student.
Δmastery = mean(mastery_after - mastery_before) per session, per KC, per student_grade
Метрика 2 · Онлайн
Retention Rate
mastery_effective через 7/14/30 дней после освоения KC. Сигнал качества decay модели и spaced repetition. Без этого decay-функцию нельзя калибровать.
retention(kc, t) = mastery_effective(t) / mastery_peak где t = days_since_mastered
Метрика 3 · Офлайн
ZPD Precision
Доля заданий из ZPD, на которых ученик ответил правильно с первой попытки. Слишком высокая (>0.9) — ZPD слишком легкий. Слишком низкая (<0.4) — слишком сложный. Оптимум 0.6–0.75.
zpd_precision = correct_first_attempt / total_zpd_tasks target: [0.60, 0.75]
Метрика 4 · Офлайн
IRT Calibration Fit
После накопления 100+ ответов per task — сравнивать предсказанную вероятность правильного ответа (по IRT) с реальной. RMSE < 0.1 — хорошая калибровка.
irt_rmse = sqrt(mean((p_irt - p_actual)^2)) per task, min 100 responses required
Метрика 5 · Сигнал качества
Time to Mastery
Сколько сессий/задач нужно ученику чтобы достичь mastery_effective >= 0.7 для KC. Ключевой business metric — система должна ускорять обучение vs baseline (линейный учебник).
ttm(kc) = sessions_until(mastery_eff >= 0.70) compare: recommendation_group vs control_group
Методология
Что делать прямо сейчас
1. Логировать все BKT updates в Kafka (mastery_before, mastery_after, kc_id, is_correct)

2. Добавить interaction.recommendation_source в схему

3. Реализовать offline replay: взять историю ответов, прогнать через BKT без рекомендации (случайные задания) — сравнить Δmastery
