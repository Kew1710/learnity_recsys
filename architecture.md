## 1. Что из себя представляет система

Рекомендательная система заданий по математике для учеников. Три задачи:
- Предлагать задания релевантные уровню знаний ученика
- Выстраивать индивидуальный учебный план
- Собирать метрики обучения

---

## 2. Архитектурный подход — Микросервисы ✅

Система строится как набор независимых **сервисов**, а не LLM-агентов. Агенты используются **только** в сервисе генерации заданий.

### Сервисы системы

| Сервис | Что делает |
|--------|-----------|
| Сервис профиля | Хранит и обновляет знания ученика (вероятности освоения тем) |
| Сервис графа знаний | Хранит дерево математических тем и их зависимостей |
| Сервис подбора тем | Ходит в граф, находит темы в зоне ближайшего развития |
| Сервис выбора задания | По формуле IRT выбирает оптимальное задание |
| Сервис банка заданий | Хранит задания с IRT-параметрами |
| Сервис генерации заданий | LLM-агент, генерирует новые задания |
| Сервис аналитики | Собирает метрики (learning gain, retention) |
| Сервис учебного плана | Долгосрочный план по темам |

---

## 3. Горячий и холодный путь ✅

### Горячий путь — синхронный

Всё что происходит **пока ученик ждёт** следующего задания (~300–500мс).

```
Ученик нажал "Проверить"
    → Gateway  POST /sessions/{id}/answer
        → Profile  POST /students/{id}/interactions  (обновить mastery)
        → Retrieval  POST /recommend
            → Profile  GET /students/{id}            (mastery + grade)
            → Profile  GET /students/{id}/seen_tasks (exclude фильтр)
            → Graph    POST /zpd                     (ZPD кандидаты)
            → TaskBank GET /tasks/by_kc              (задания для KC)
            → select_kc_from_zpd + select_task       (exploitation/exploration)
    → Вернуть {mastery_update, next_task} ученику
```

Общение: **синхронное (gRPC или REST)**.

### Холодный путь — асинхронный

Всё что можно сделать **после** того как ученик получил задание, в фоне.

```
Событие "Ученик ответил на задание" →
    → Сервис аналитики (сохранить метрики)
    → Сервис калибровки IRT (обновить параметры задания)
    → Сервис учебного плана (скорректировать план)
    → Сервис забывания (пересчитать decay по расписанию)
```

Общение: **асинхронное через Kafka**.

## 4. Способ общения сервисов ✅

### Синхронное общение (горячий путь)
- gRPC внутри кластера — быстрее REST, строгие контракты через Protobuf
- REST — как альтернатива, проще в начале

### Асинхронное общение (холодный путь) — **Kafka** ✅

**Почему Kafka, а не RabbitMQ:**
- Kafka хранит историю событий долго (дни, недели) — можно перечитать
- Позволяет пересчитать профили учеников заново с нуля по истории
- Ценно для переобучения моделей и аудита
- RabbitMQ удаляет сообщения после обработки — история теряется

**Принцип:** один сервис публикует событие, несколько потребителей читают независимо.

## 5. Известные проблемы (из анализа исходной архитектуры)

### Критичные (блокеры)
- **Cold Start** — нет модуля диагностики для нового ученика
- **Валидация LLM-заданий** — LLM делает математические ошибки, нужна проверка (SymPy)
- **IRT параметры для новых заданий** — у сгенерированных задач нет параметров a/b/c

### Важные
- **Алгоритм обновления знаний** — BKT слишком примитивен, DKT требует данных; рассмотреть DKVMN
- **Кривая забывания** — нужна полноценная реализация (FSRS), а не просто timestamp
- **Exploration** — чистый IRT это exploitation; нужен bandit (Thompson Sampling)
- **Граф знаний** — кто строит, как обновляет, какая схема

### Средние
- **Макро-планирование** — нет долгосрочного учебного плана
- **Evaluation pipeline** — как измерять что система работает

## 6. Хранение профиля ученика ✅

**MVP (10-30 учеников): только PostgreSQL.**

При таком масштабе производительность не является проблемой. PostgreSQL справляется без кеша.

### Схема таблиц

```
students          → базовые данные ученика
mastery           → (student_id, kc_id, probability, last_practiced) — отдельная таблица, не JSON
interactions      → вся история ответов (student_id, task_id, is_correct, timestamp, ...)
```

### Правила которые закладываем сразу

- **Только сервис профиля пишет** `mastery` — остальные сервисы только читают
- **Индексы** на `student_id` и `kc_id` с самого начала
- **`mastery` как отдельная таблица** (не JSON-колонка) — чтобы легко добавить Redis поверх позже

### Путь миграции при росте

```
MVP (10-30 учеников):   PostgreSQL
При росте (100+):        Redis (кеш mastery) + PostgreSQL (источник правды)
При большом росте:       CQRS поверх Redis + PostgreSQL
```

## 7. Структура графа знаний ✅

Граф — это DAG (направленный ациклический граф), не дерево. Одна нода может иметь несколько пререквизитов из разных разделов.

### Уровни нод

```
Предмет  →  Раздел  →  Тема  →  KC (микро-навык)
```

Рекомендательная система работает на уровне **KC**. Разделы и темы — для навигации.

### Схема ноды (KC)

```python
{
  # Идентификация
  "id": "kc_pythagorean_theorem_apply",
  "name": "Применение теоремы Пифагора",
  "level": "kc",                    # subject | section | topic | kc
  "subject": "Математика",
  "section": "Геометрия",
  "topic": "Теорема Пифагора",

  # Для рекомендательной системы
  "difficulty_base": 0.6,

  # Для генерации заданий (контекст для LLM)
  "learning_objectives": [...],
  "key_formulas": ["a² + b² = c²"],
  "common_misconceptions": [...],
  "generation_constraints": "..."
}
```

### Схема ребра

```python
{
  "from": "kc_sqrt_compute",
  "to": "kc_pythagorean_theorem_apply",
  "type": "prerequisite",
  "strength": 0.9    # 0.0–1.0, насколько критичен пререквизит
}
```

### Правило атрибуции ошибок (закреплено в схеме)

**Проблема**: если ученик ошибается в задаче на несколько тем и даёт один числовой ответ — неизвестно в какой теме ошибка. BKT получает неточный сигнал.

**Решение**: задание с >2 уникальными `primary_kcs` суммарно **обязано быть многочастным** (`parts > 1`). Каждая часть тестирует конкретную KC и даёт точный BKT-сигнал. Это правило валидируется на уровне Pydantic-схемы `Task` — создать некорректное задание невозможно.

```
Допустимо:    1 часть, 1–2 primary KC   → ответ один, KC понятна
Допустимо:    N частей, любое число KC  → каждая часть = отдельный сигнал
Запрещено:    1 часть, >2 primary KC    → ошибку атрибутировать нельзя
```

Ученик ошибся в части 1 → негативный BKT-сигнал только для KC части 1, остальные части обновляются независимо.

### Кто строит граф

В будущем: пользователь (через интерфейс) + LLM (генерирует ноды).

**Сейчас для разработки рекомендательной системы:** граф используется через чёткий API — системе всё равно кто его наполнял. Реальное наполнение откладывается до написания тестов (seed-скрипт с ~20-30 KC по одной теме).

### API сервиса графа (контракт)

- `get_node(kc_id)` — получить ноду
- `get_prerequisites(kc_id)` — получить пререквизиты
- `get_zpd_candidates(mastery_probabilities)` — получить KC в зоне ближайшего развития
- `get_path(from_kc, to_kc)` — найти путь между темами

## 9. Cold Start / Placement Assessment ✅

### Класс ученика — фильтр на граф

`grade` в профиле ученика ограничивает видимые KC-ноды. Каждая KC-нода хранит `grade_introduced`.

```
Жёсткий пол:     grade_introduced < student.grade - 2  →  не показывать
Основная зона:   grade_introduced <= student.grade      →  основная работа
Зона опережения: grade_introduced == student.grade + 1  →  доступно при высоком mastery
```

Класс — мягкое ограничение, не потолок. Продвинутый ученик может опережать программу.

### Схема Cold Start (MVP)

1. **Регистрация** — ученик выбирает класс
2. **Инициализация профиля** через `POST /students/{id}/mastery/cold_start`:
   - KC из класса `n-3` и ниже → `mastery = 1.0` (освоено давно, не трогаем)
   - KC из класса `n-2` → `mastery = 0.8`
   - KC из класса `n-1` → `mastery = 0.5`
   - KC текущего класса и выше → не инициализируем (`mastery = 0.0` по умолчанию)
3. **Входной тест** — заглушка в API на будущее. Когда будет реализован — корректирует `mastery` по реальным ответам.

### Что добавляется в схему

```python
# Профиль ученика
{ "grade": 8, ... }

# KC-нода
{ "grade_introduced": 8, ... }
```

## 10. Алгоритм обновления знаний ✅

### Базовый алгоритм — BKT (Bayesian Knowledge Tracing)

Работает с малым количеством данных, интерпретируемый. DKT/DKVMN — после набора тысяч студентов, интерфейс сервиса не изменится.

Четыре параметра на каждую KC:
- `P(L0)` — начальная вероятность знания
- `P(T)` — вероятность выучить за одну попытку
- `P(S)` — вероятность ошибиться зная KC (slip)
- `P(G)` — вероятность угадать не зная KC (guess)

### Расширения поверх BKT

**Забывание (Decay):**
```python
decay_factor = 0.5 ** (days_since / half_life_days)
mastery_effective = mastery_stored * decay_factor
```
В базе хранится `mastery_stored`. Для подбора задания используется `mastery_effective`.

`half_life_days` задаётся на уровне KC в графе знаний, по предмету:

| Предмет | half_life_days | Обоснование |
|---------|---------------|-------------|
| arithmetic | 90 | процедурные навыки держатся долго |
| geometry | 60 | геометрия требует периодического повторения |
| algebra | 45 | абстрактные концепты забываются быстрее |
| statistics | 30 | редко используется вне учёбы |

**Режим активного освоения (`review_mode`):**

Флаг на уровне ученика. Переключается через `PATCH /students/{id}/review_mode`.

- `review_mode = False` (по умолчанию) — decay применяется нормально
- `review_mode = True` — decay пропускается; система считает что всё помнится и не отвлекает на прошлые темы

Используется когда ученик активно проходит программу и уверен в предыдущем материале.

**Подсказки снижают вес обновления:**
```python
update_weight = 0.4 if hints_used else 1.0
```

**Несколько KC в задании:**
```python
primary_kcs   → weight = 1.0
secondary_kcs → weight = 0.3
```

### Частичный ответ (небинарность)

Задание может иметь несколько частей (`parts[]`). Каждая часть — отдельная KC, отдельный BKT update.

Внутри части — `score: float [0.0–1.0]` вместо бинарного `correct`. BKT модифицирован для работы с непрерывным score.

Кто выставляет score:
- Числовой ответ → автоматически
- Доказательство / текст → учитель вручную (MVP)

### Порядок обновления

```
Получили ответ
  → Применить decay (с момента last_practiced)
  → BKT update per part, per KC (с учётом score и hints_used)
  → Обновить mastery + last_practiced в БД
  → Опубликовать событие в Kafka
```

## 11. Схема данных ✅

### Задание (Task)

```
Task:
  task_id:        UUID
  grade_min:      int
  source:         "bank" | "generated"
  created_at:     datetime

  parts: [
    Part:
      part_id:            str
      description:        str

      primary_kcs:        List[str]          # обязательно, минимум одна KC
      secondary_kcs:      List[str] = []     # опционально

      answer_type:        "numeric" | "multiple_choice" | "multi_select" | "proof"
      correct_answer:     Any | None         # None для "proof"
      tolerance:          float | None       # только для "numeric"

      irt:                IRT | None         # None для новых заданий без калибровки
        difficulty:       float
        discrimination:   float
        guessing:         float

      scaffolding_steps:  List[str] = []
      distractors_map:    Dict[str, str] = {}    # неправильный ответ → misconception_id
  ]
```

### Профиль ученика (Student)

```
Student:
  student_id:   UUID
  grade:        int
  created_at:   datetime

  behavioral:
    guessing_rate:      float
    hint_dependence:    float

  mastery: Dict[kc_id →
    MasteryRecord:
      probability:    float
      last_practiced: datetime
      attempts_count: int
      p_transit:      float
      p_slip:         float
      p_guess:        float
  ]
```

### Профиль ученика (Student) — дополнительные поля

```
  review_mode:  bool = False   # True → decay отключён, активное освоение
```

### Лог взаимодействий (Interaction)

```
Interaction:
  interaction_id:           UUID
  student_id:               UUID
  task_id:                  UUID
  part_id:                  str
  score:                    float       # 0.0–1.0
  hints_used:               int
  time_spent_seconds:       int
  timestamp:                datetime
  misconception_triggered:  str | None
  recommendation_source:    str | None  # "zpd" | "exploration" | "placement" | "manual"
```

`recommendation_source` — ключевое поле для eval: позволяет сравнивать качество разных стратегий рекомендации.

Ответ `POST /interactions` возвращает `mastery_before` для каждой KC — основа для вычисления learning gain.

### Хранение

```
PostgreSQL:
  tasks, parts            → задания
  students                → профиль и behavioral
  mastery                 → (student_id, kc_id, все поля MasteryRecord)
  interactions            → лог ответов

Вычисляется на лету:
  mastery_effective = mastery.probability * decay_factor
```

## 12. Технологический стек ✅

| Слой | Решение |
|------|---------|
| Язык | Python |
| Фреймворк сервисов | FastAPI |
| ORM | SQLAlchemy |
| Основная БД | PostgreSQL |
| Граф знаний | Neo4j |
| Очередь событий | Kafka (KRaft mode, без Zookeeper) |
| Локальный запуск | Docker Compose |

### Стратегия локальной разработки

**Инфраструктура в Docker Compose** (PostgreSQL, Neo4j, Kafka) — не пересобираются при изменении кода.

**Сервисы запускаются локально** через `uvicorn main:app --reload` — мгновенный перезапуск при изменении файла.

**Полный стек** (`docker-compose.full.yml`) — поднимается когда нужно проверить интеграцию всех сервисов вместе.

### Структура проекта

```
learnity/
  docker-compose.yml           # инфраструктура (postgres, neo4j, kafka)
  docker-compose.full.yml      # полный стек для интеграционного тестирования
  services/
    profile/                   # сервис профиля
    graph/                     # сервис графа знаний
    task_bank/                 # сервис банка заданий        ✅
    retrieval/                 # сервис подбора тем           ✅
    ranking/                   # сервис выбора задания        (встроен в retrieval)
    gateway/                   # API-шлюз (горячий путь)     ✅
  shared/
    schemas.py                 # общие Pydantic-схемы (Task, Student, ...)
    kafka_client.py            # общий Kafka клиент

```

### Путь к production

```
Локально:   Docker Compose (инфраструктура) + uvicorn локально
Staging:    Docker Compose полный стек
Production: Kubernetes (когда масштаб потребует)
```

## Отложено

- **Сервис генерации заданий (LLM)** — после того как рекомендательная система работает стабильно
- **IRT калибровка новых заданий** — зависит от сервиса генерации
- **Evaluation pipeline** — метрики качества (learning gain, retention); инфраструктура заложена (recommendation_source, mastery_before), агрегация — после первых учеников
- **Exploration / Bandit** — Thompson Sampling поверх IRT, после накопления данных
- **Redis кеш** — когда учеников станет 100+
